  Data leakage: pipeline is sound

  Verified directly: triple-barrier labeling only looks at bars after entry
  (path = data.iloc[pos+1:end_pos+1]), feature engineering uses only causal
  log-returns/EMAs (no center=True, no negative shifts), scalers are fit only on
  sampled train data, and prepare.py explicitly drops any sample whose
  label-exit timestamp crosses a split boundary (leakage_report.csv). This part
  is well engineered.

  New bug I found (not previously caught)

  src/kvant/training/pytorch_trainer.py:149 — n_classes = int(y_train.max()) + 1
  is inferred from the training labels actually present in a fold, but every
  model (resnls.py, conv1d.py, etc.) is hardcoded to n_classes=3 at construction
  time, before fit() runs. If a fold's train split happens to contain zero
  samples of class 2 (BUY) — plausible given how imbalanced labels are in this
  pipeline — _class_weights builds a length-2 weight tensor while the net still
  outputs 3 logits, and CrossEntropyLoss(weight=...) will throw a shape-mismatch
  crash mid-walk-forward. Fix: pass n_classes=model.n_classes explicitly
  instead of inferring it.

  Carried over from your prior sessions — still unresolved in the code today

  1. pipeline.toml: gap_days = 0 — was flipped to 1 then reverted back to 0
  because it broke fold generation with the current short data window. Still
  technically allows label look-ahead across fold boundaries (mitigated in
  practice since folds sit at midnight and width_minutes=40, but it's not
  actually purged).
  2. Cross-boundary sample exclusion (prepare.py _build_index) — silently drops
  every trade entered in the last width_minutes of a fold's test window, causing
  the equity curve to flatline near the end of each fold. A fix was proposed
  (let the labeler see a buffer past test_end_exclusive) but never applied.
  3. allow_short = false force-converts every SHORT call to HOLD post-hoc, which
  was capping evaluated accuracy near 20% even though the raw model's
  directional accuracy was ~50%. This is a config/policy choice, not a bug, but
  it's worth knowing it's still set this way since it makes the confusion matrix
  misleading.
  4. attention_tsb.py:73 — still only uses the final timestep after attention,
  which defeats most of the purpose of self-attention vs a plain LSTM.
  Mean-pooling was suggested but not applied.
