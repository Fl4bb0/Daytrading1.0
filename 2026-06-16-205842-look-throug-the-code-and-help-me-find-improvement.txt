● Here's a comprehensive breakdown of what I found. I'll separate genuine
  bugs/weaknesses from feature ideas.

  ---
  Bugs & Concrete Weaknesses

  1. Raw OHLC prices are non-stationary — biggest impact

  feature_engineering.py:136–139

  OHLC prices go directly into the model as raw levels. Price levels drift over
  months, so a model trained in January will see completely different input
  scales in July. Use log-returns instead:

  # Instead of raw open/high/low/close:
  ohlc["close_ret"] = np.log(close / close.shift(1))
  ohlc["hl_range"]  = np.log(df["high"] / df["low"])          # intrabar range
  ohlc["co_ret"]    = np.log(close / df["open"])               # bar body
  ohlc["ho_ret"]    = np.log(df["high"] / df["open"])
  ohlc["lo_ret"]    = np.log(df["low"]  / df["open"])
  This is the single highest-impact fix for generalization across walk-forward
  folds.

  ---
  2. Early stopping monitors accuracy on imbalanced classes

  pytorch_trainer.py:176

  metric = val_acc if val_loader is not None else -train_loss

  With HOLD dominating the label distribution, a model that predicts HOLD 80% of
  the time gets high accuracy but generates no trades. Early stopping should
  monitor macro-F1 or at minimum balanced accuracy. Otherwise you'll
  consistently get the "best" checkpoint being one that over-predicts HOLD.

  ---
  3. RSI division by zero is unguarded

  feature_engineering.py:133

  rs = avg_gain / avg_loss  # avg_loss can be 0 → inf → NaN propagates
  When there are no down-moves in a window (strong uptrend), avg_loss is 0,
  producing inf. That NaN propagates silently through fillna(0) into the model.
  Add avg_loss = avg_loss.replace(0, 1e-10) before the division.

  ---
  4. No gradient clipping — LSTM can explode

  pytorch_trainer.py:243–246

  loss.backward()
  optimizer.step()   # ← no clip_grad_norm
  LSTMs are prone to gradient explosion, especially with long sequences. Add
  before optimizer.step():
  torch.nn.utils.clip_grad_norm_(net.parameters(), max_norm=1.0)

  ---
  5. walk_forward.py:152 raises SystemExit in library code

  Should be raise ValueError(...). SystemExit bypasses try/except blocks and
  makes it impossible to catch from calling scripts.

  ---
  6. gap_days = 0 in walk-forward config

  pipeline.toml:47

  The code comment says "1 day is a safe purge buffer" for the 40-minute
  triple-barrier. With gap_days = 0, labels from the end of the training window
  can overlap with the start of the val window, introducing look-ahead leakage.
  Set to at least 1.

  ---
  7. n_classes inferred from label max — fragile

  pytorch_trainer.py:145

  n_classes = int(y_train.max()) + 1
  If class 2 (BUY) is absent from a small training fold, the model gets
  initialized with 2 output classes instead of 3. Pass n_classes explicitly from
  the model.

  ---
  Architecture Improvements

  8. attention_tsb wastes its attention mechanism

  attention_tsb.py:73

  return self.head(attended[:, -1, :])  # only last timestep
  After re-weighting all timesteps via attention, you discard all but the last
  one — same as a plain LSTM. Use mean pooling or a learned query:
  return self.head(attended.mean(dim=1))

  ---
  9. Meta model reports in-sample metrics only

  meta/model.py:60–65 and meta/model.py:163–165

  Both RidgeMetaModel.fit() and BinaryMetaClassifier.fit() evaluate on the exact
  data they trained on. These numbers look great but tell you nothing about
  real performance. Add at minimum a k-fold cross-val score during training.

  ---
  Features Worth Adding

  10. Focal loss instead of weighted CrossEntropy

  Weighted CE reduces HOLD dominance in the loss but still treats all hard/easy
  examples equally. Focal loss downweights easy examples the model already
  predicts correctly, forcing it to focus on the rare SHORT/BUY signals. Drop-in
  replacement in pytorch_trainer.py:147:
  # pip install focal-loss-torch  (or implement inline)
  criterion = FocalLoss(gamma=2.0, weight=weights)

  11. Bollinger Band position + ATR

  Two features with high signal-to-noise for intraday:
  - bb_pct = (close - BB_lower) / (BB_upper - BB_lower) — where in the band the
  price sits
  - atr_14 = rolling ATR — used as a dynamic barrier size and regime indicator

  These would integrate cleanly into IntradayTA10Features._transform_df().

  12. Probability calibration before meta filtering

  The raw softmax probabilities from the model are not well-calibrated
  (overconfident). Before feeding side_confidence into the meta model, apply
  temperature scaling or Platt scaling. A miscalibrated side_confidence makes
  the meta score unreliable, since it's the most important meta feature.

  13. Sharpe / Sortino in evaluate()

  The evaluator produces equity curves but no risk-adjusted metrics. Adding
  Sharpe and Sortino to evaluation/runner.py would let the walk-forward
  comparison rank folds by risk-adjusted return, not just raw accuracy.

  ---
  Top 3 to do first: raw OHLC → returns (#1), early stopping metric (#2),
  gradient clipping (#4). These are the changes most likely to immediately
  improve real walk-forward test performance. Want me to implement any of these?

  With gap_days = 1:

  Fold 0: train→05-01 | val 05-01→06-01 | test 06-01→07-01 ✓

● The HOLD scarcity traces directly to two things, one pre-existing and one I
  just changed.

  1. Class-weighted loss (pytorch_trainer.py:73–78, pre-existing)

  def _class_weights(y, n_classes):
      counts = np.bincount(y, minlength=n_classes)
      w = counts.sum() / counts        # inverse frequency
      return w / w.mean()

  HOLD is almost certainly your majority class (triple-barrier vertical exits
  happen far more often than barrier hits within 40 minutes). Full
  inverse-frequency weighting means the loss penalty for missing a BUY/SHORT is
  much larger than for misfiring one. The optimizer responds by pushing the
  decision boundary to favor predicting BUY/SHORT more readily — it's cheaper in
  loss terms to over-trade than to sit out a real signal.

  2. Macro-F1 checkpoint selection (pytorch_trainer.py:171, the change I made 
  last)

  Macro-F1 averages per-class F1 equally regardless of frequency. A checkpoint
  with strong BUY/SHORT recall (even with mediocre precision — i.e.
  trigger-happy) scores better on macro-F1 than one that mostly sits in HOLD. So
  switching early stopping from accuracy → macro-F1 made this directional bias
  more pronounced, not less — it's now actively selected for.

  The required_buy_probability/required_sell_probability = 0.5 gate
  (pipeline.toml) catches some of this by demoting low-confidence directional
  calls back to HOLD, but it only helps if softmax probabilities are
  well-calibrated — and with class-weighted CE, they typically aren't
  (probabilities run systematically high for the up-weighted classes).

  Fixes, in order of impact:

  1. Dampen the class weights — use sqrt(inverse frequency) instead of full
  inverse frequency, so minority classes are favored without being over-favored.
  2. Raise the probability thresholds in pipeline.toml
